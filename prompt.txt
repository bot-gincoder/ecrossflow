
Implémente un système multilingue complet, robuste, SEO-friendly et extensible pour le site.

OBJECTIF
Je veux que le site soit traduisible automatiquement dans un grand nombre de langues, tout en gardant une architecture propre, rapide, maintenable et professionnelle.
Je ne veux pas d’un simple widget de traduction navigateur.
Je veux une vraie implémentation i18n cté application, avec URLs par langue, fallback propre, traduction automatique cté serveur, et possibilité de corriger les traductions manuellement.

ARCHITECTURE CIBLE
1. Utiliser une architecture i18n native basée sur le routing par locale.
2. Chaque langue doit avoir sa propre URL, par exemple :
   /fr
   /en
   /es
   /pt
   /de
   etc.
3. Conserver une langue source principale pour tout le projet : FR par défaut.
4. Séparer strictement :
   - les textes UI (menus, boutons, labels, messages d’erreur)
   - le contenu métier / marketing / SEO
   - le contenu dynamique venant de la base de données
5. Mettre un système de fallback :
   - si une traduction est absente dans une langue, afficher la valeur FR
   - ne jamais casser l’affichage
6. Prévoir un sélecteur de langue visible et propre.
7. Détecter automatiquement la langue préférée du visiteur à la première visite, puis mémoriser son choix.
8. Générer automatiquement les traductions via une API serveur sécurisée.
9. Prévoir une couche d’édition manuelle pour corriger les traductions importantes.
10. Le système doit être scalable pour ajouter beaucoup de langues plus tard.

STACK RECOMMANDÉE
Si le projet est sous Next.js, utilise l’App Router avec une vraie structure locale. Next.js documente officiellement l’internationalisation du routing ainsi que l’App Router. 0

TRADUCTION AUTOMATIQUE
Utiliser Google Cloud Translation cté serveur pour :
- lister les langues supportées
- traduire automatiquement les nouvelles chanes
- étendre facilement à beaucoup de langues
Google Cloud documente le listage des langues supportées et la traduction programmatique. 1

IMPORTANT
- Aucune clé API ne doit être exposée cté client
- Toute traduction automatique doit passer par une route serveur / backend
- Les résultats doivent être stockés, pas recalculés à chaque chargement
- Les pages importantes doivent pouvoir être revues manuellement
- Prévoir plus tard un glossaire métier / custom translation pour les termes de marque ou expressions métier sensibles. Google Cloud documente aussi les traductions personnalisées pour adapter style et vocabulaire. 2

STRUCTURE TECHNIQUE ATTENDUE

A. ROUTING
Créer une structure de routes localisées.
Exemple :
app/
  [locale]/
    layout.tsx
    page.tsx
    about/page.tsx
    contact/page.tsx

Le locale doit être validé via une liste centralisée.
Le layout doit charger le dictionnaire de traduction correspondant.

B. DOSSIER DE TRADUCTIONS
Créer une structure du type :
src/
  i18n/
    config.ts
    get-dictionary.ts
    locales/
      fr/
        common.json
        home.json
        pricing.json
      en/
        common.json
        home.json
        pricing.json
      es/
        common.json
        home.json
        pricing.json

C. CONFIG I18N
Créer un fichier central qui contient :
- locales supportées
- locale par défaut
- langues RTL éventuelles
- mapping nom affichable / code langue
- fallback locale

D. CHARGEMENT DES TRADUCTIONS
Créer une fonction serveur du type :
- getDictionary(locale, namespace)
ou
- getMessages(locale)

Elle doit :
- charger la bonne langue
- fallback sur FR si clé absente
- supporter le découpage par namespace

E. UTILISATION DANS LES COMPOSANTS
Aucun texte visible ne doit rester hardcodé dans les composants UI.
Remplacer par des clés du type :
t("hero.title")
t("hero.subtitle")
t("nav.features")
t("auth.login")
t("errors.network")

F. LANGUAGE SWITCHER
Créer un composant de changement de langue :
- accessible
- responsive
- garde la route actuelle si possible
- met à jour l’URL avec la locale
- sauvegarde la préférence dans cookie ou local storage
- fonctionne sans casser SEO

G. DÉTECTION AUTO DE LANGUE
À la première visite :
- lire Accept-Language ou préférence sauvegardée
- rediriger vers la locale la plus proche supportée
- sinon fallback FR

H. SEO MULTILINGUE
Implémenter :
- URLs par langue
- balises hreflang
- canonical correct par version
- metadata traduites par locale
- sitemap localisé si possible
- titres et descriptions SEO par langue

I. CONTENU DYNAMIQUE
Pour tout contenu en base :
Créer une structure de traduction de type :
translations:
  entity_type
  entity_id
  field_name
  locale
  value
  status
  source_hash
  updated_at

Ou une structure équivalente propre.

Exemple :
Page
- id
- slug
- source_locale
- status

PageTranslation
- page_id
- locale
- title
- description
- body
- seo_title
- seo_description
- is_auto_translated
- is_reviewed
- updated_at

J. PIPELINE DE TRADUCTION AUTOMATIQUE
Créer un pipeline serveur qui :
1. détecte les nouvelles chanes source
2. calcule un hash du texte source
3. vérifie si une traduction existe déjà
4. sinon appelle l’API de traduction
5. stocke le résultat
6. marque la traduction comme auto-generated
7. permet ensuite une correction manuelle

K. ADMIN INTERNE
Créer une base pour un mini backoffice de traduction avec :
- liste des langues
- liste des clés manquantes
- statut : auto / reviewed / missing
- édition manuelle d’une traduction
- filtre par page / namespace / langue
- recherche par clé

L. PERFORMANCE
Le site doit :
- charger uniquement les messages nécessaires
- éviter de charger toutes les langues en même temps
- mettre en cache les dictionnaires
- éviter toute traduction API en runtime cté client
- rester rapide sur mobile

M. ERREURS ET FALLBACKS
Prévoir :
- logs propres si une clé manque
- fallback FR
- aucune page blanche
- aucun crash si une langue n’est pas encore complète

N. RTL
Prévoir le support RTL pour les langues comme l’arabe :
- dir="rtl" automatiquement
- styles compatibles
- inversion logique si nécessaire

O. SÉCURITÉ
- secret de traduction uniquement cté serveur
- variable d’environnement sécurisée
- aucune fuite de token dans le bundle client
- rate limiting sur les endpoints de traduction si exposés en interne

VARIABLES D’ENVIRONNEMENT
Prévoir au minimum :
GOOGLE_CLOUD_PROJECT_ID=
GOOGLE_CLOUD_LOCATION=
GOOGLE_APPLICATION_CREDENTIALS=
DEFAULT_LOCALE=fr
